## Git & github

<h1>GIT</h1>

<p><strong> GIT </strong> — это распределённая система управления версиями. Она используется для отслеживания изменений в исходном коде программного обеспечения. GIT позволяет разработчикам сохранять историю изменений, работать над разными версиями проекта одновременно, и сотрудничать с другими разработчиками, синхронизируя изменения. Это важный инструмент в командной разработке и для контроля качества кода.</p>

[Приложение бесплатное качаем тут](https://git-scm.com/)

![Скрин Git Bash](/img/Скрин%20внутри%20Gita.png)

<ol>
<li>Нам нужно задать имя пользователя под которым мы будем вносить все изменения в файлах за которыми следит GIT чтобы он таким образом мог запомнить кто именно эти файлы меняет такие настройки задаем с помощью команды.</li>

<strong>global</strong> - обозначает что эти настройки будут применяться к любому файлу, который вы отдадите под контроль.

</ol>

<h2> Create repository </h2>

<p> <strong> Репозиторий </strong> - папка с файлами за которой у вас на компьютере следит Git.</p>

Репозиторий может быть <strong>локальный</strong> и <strong>удаленный</strong>:

<strong>Локальный</strong> - папка с файлами, которая храниться именно на вашем компьютере за которой следит Git.

<strong>Удаленный</strong> - по сути тот же локальный, но вы взяли и с помощью <em>Gita</em> загрузили его на <em>GitHub</em> и теперь он хранится там. По сути тот же локальный только он хранится не на вашем компьютере, а в облаке тоесть удаленно.

<p>Для того чтобы нам клонировать репозиторий локально нам нужно перейти в <em>Git Bash</em> и писать:
<ol>

<li> используем команду <strong>git clone</strong> с (url удаленного репозитория) - скачивает все файлы из удаленного репозитория, создавая вашу локальную копию этого репозитория.</li>
</ol>

<h2>Внесение изменений в локальном репозитории</h2>

<p>Теперь поговорим о внесении изменений в вашем локальном репозитории.</p>

<p>Для того чтобы понять в каком состоянии <em>Git</em> видит наши файлы внутри репозиторий выполняем команду <strong>git status</strong> - показывает текущий статус всех несохраненных (uncommited) изменений файлов в репозитории.</p>

<strong> git commit -m "sms"</strong> - <em>команда для коммитит</em> так же в "" можно оставить сообщение какой это коммит.

<strong> git add + какой то файл </strong> - <em>команда</em>, которая позволяет коммитить определенный файл либо папку, подготавливает данные файлы к следующему коммиту, добавляя их в <em>"индекс Gita"</em>.

<strong>"Индекс Gita"</strong> - <em>это хранилище файлов</em>, готовых к коммиту.

<strong> git add . </strong> - <em>команда для коммита</em>, точка показывает то что мы хотим закомитить все файлы из текущей директории.

<strong> git log </strong> - <em>показывает историю всех коммитов в данной ветке с их хэшами и сообщениями</em>.

<strong> git push <имя репозитория куда вы хотите загрузить изменения origin> <ветка изменений куда мы загружаем коммиты main> </strong> - <em>публикует коммиты из вашего локального репозитория в удаленный с данным именем в данной его ветке.</em>

<strong> git pull <имя репозитория origin> <ветка из которой мы подгружаем коммиты main> </strong> - <em> скачивает из данного удаленного репозитория все коммиты в данной ветке, которых еще нет в локальном репозитории</em>.

<h1> Branches </h1>
<p>Ветки - последовательность коммитов, которое имеет определенное имя.</p>
<p>Главная ветка называется <strong>master</strong> либо <strong>main</strong>.</p>
<p>Веток можно создавать до бесконечности, вы спросите зачем нам ветки?<br> Ответ прост представьте что вы разработчик и работаете над каким то проэктом и делаете огромную фичу там очень много файлов, которые нужно изменить, вы их меняете, постоянно делаете новые коммиты чтобы не терять изменения, НО еше не закончили и все еще в процессе создания фичи, и вы не хотите чтобы ваши изменения попадали в общий репозиторий (main), и не хотите чтобы люди их видели потому что они все еще не закончены и могут быть баги в незаконченной фиче поэтому мы и должны создать ветку изолированную от maina простыми словами создать ветку с названием "my project" и рабоать в ней чтобы остальные разрабы работали с мейном и ваши коды не пересекались.</p>


<p>тобы посмотреть какие ветки в репозитории есть используем команду - <strong>git branch</strong>

<strong> git branch </strong> - <em>выводит список локальных веток</em>, доступных прямо сейчас в данном локальному репозитории и звездой <strong>(*)</strong> будет отмечаться та ветка в которой вы находитесь.

![Скрин звезды ](/img/Скрин%20звезды%20в%20бранче.png)

<strong>git branch <имя ветки пример:"myproject"> </strong> - <em>команда чтобы создать новую ветку.</em>
<p>Когда мы создаем новую ветку, то по умолчанию она создается на основе последнего коммита той ветки на которой мы находились в момент создания.</p>

<strong>git checkout <имя ветки> </strong> - <em>команда чтобы переключаться с ветки на ветку.</em>
<p>Чтобы опубликовать изменения с локального <strong>(VScode либо другая программа)</strong> в удаленный <strong>(GitHub)</strong>, то переходим сначала на ту ветку с которой хотим изменения опубликовать и используем команду <strong>git push origin <название ветки "my project"> </strong>, которую публикуем </p>

<h2>Слияние (merge) </h2>
<p>Представьте ситуиацию что вы работали вот над этой большой фичей в отдельной ветке и делали коммиты и наконец довели эту фичу до ума и хотите перенести все ваши изменения загрузить на основную ветку <strong>(main)</strong> такая процедура в <strong>Git</strong> называется <em>слияние веток</em> есть <strong>2</strong> варианта как это сделать:

1) <strong>при помощи merge </strong> 
<ol>
<li> сначала мы должны перейти на главную ветку командой git checkout main </li>
<li>выполнить команду <strong>git merge</strong> с именем ветки из которой мы будем проводить слияние <em>"my project"</em> </li>
<li>если вы хотите опубликовать merge, то должны выполнить <strong>git push origin main</strong> </li>
</ol>

<p><strong>Git</strong> берет все изменения с переданной ветки и <em>добавляет все их в ветку main в виде 1 коммита</em>. Убедиться в этом легко просто нужно ввести команду <strong>git log и вы увидите этот коммит</strong>.</p>

<p><strong>Нужно помнить что merge по факту создает 1 коммит в main, но если вы хотите поддержать историю изменений и перевести все коммиты с рабочей ветки в main, то для вас подходит 2 способ!</strong> </p>

2) <strong>при помощи git rebase <имя ветки> </strong> - <em> перемещает коммиты из данной ветки в ту, на которой сейчас находится пользователь.</em>
<ol>
 <li>сначала мы должны перейти на главную ветку командой <strong>git checkout main </strong> </li>
 <li>выполнить команду <strong>git rebase с именем ветки из которой мы будем переносить коммиты "my project" в наш main </strong>
 <li> >если вы хотите опубликовать, то должны выполнить <strong>git push origin main</strong> </li>
</ol>

<strong>Если вам нужна история коммитов в ветке, то используйте rebase!</strong>
